# 테스트 전략 가이드

## 개요

AI가 코드를 생성하거나 수정할 때 따라야 할 테스트 원칙과 전략을 정의합니다.

## 핵심 원칙

- **[HARD] Reproduction-First**: 버그 수정 전 반드시 실패하는 재현 테스트를 먼저 작성한다.
- **[HARD] Test Before Merge**: 새 기능 구현 시 테스트 없이 완료 처리하지 않는다.
- 테스트는 구현 세부사항이 아닌 **동작(behavior)** 을 검증한다.
- 테스트 코드도 프로덕션 코드와 동일한 품질 기준을 적용한다.

## 테스트 레이어

### 단위 테스트 (Unit Test)
- 단일 함수/클래스의 독립적 동작 검증
- 외부 의존성은 모두 mock/stub 처리
- 실행 속도: 빠름 (ms 단위)
- 커버리지 목표: 핵심 비즈니스 로직 80% 이상

### 통합 테스트 (Integration Test)
- 모듈 간 인터페이스 및 데이터 흐름 검증
- 실제 DB, 파일 시스템 등 일부 외부 의존성 허용
- 실행 속도: 중간
- 주요 흐름(happy path + error path) 중심으로 작성

### E2E 테스트 (End-to-End Test)
- 사용자 관점의 전체 시나리오 검증
- 실제 환경과 동일한 조건에서 실행
- 핵심 시나리오만 선별하여 유지비용 최소화

## 테스트 작성 기준

### 파일 구조
- 테스트 파일은 대상 파일과 동일한 경로에 위치하거나 `tests/` 폴더로 분리
- 언어별 관행을 따름 (예: Go는 `_test.go`, Python은 `test_*.py`, TS는 `*.spec.ts`)
- 자세한 언어별 명령어는 `@.ai/rules/language/` 참조

### 테스트 케이스 명명
```
[대상 함수/기능] + [시나리오] + [기대 결과]
예: "calculateTotal_withDiscount_returnsReducedPrice"
예: "login_withInvalidPassword_throws401Error"
```

### Mock/Stub 사용 기준
- 외부 API, DB, 파일 I/O → 항상 mock
- 같은 서비스 내부의 다른 모듈 → 통합 테스트에서는 실제 사용 권장
- 시간/날짜 의존 로직 → 항상 mock

## 테스트 커버리지

커버리지 수치 기준은 `@.ai/config/quality.yaml`을 참조합니다.

- 커버리지 목표치 달성이 목적이 아닌, 중요한 경로가 실제로 테스트되는지가 핵심
- 커버리지가 높아도 assertion이 부실하면 의미 없음

## 체크리스트

- [ ] 새 기능에 단위 테스트가 있는가?
- [ ] 버그 수정 시 재현 테스트가 먼저 작성되었는가?
- [ ] 엣지 케이스(빈 값, null, 경계값)가 테스트되었는가?
- [ ] 테스트가 독립적으로 실행 가능한가? (순서 의존성 없음)
- [ ] 테스트 실행 후 외부 상태를 정리(teardown)하는가?
